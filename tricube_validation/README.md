# TriCube Validation Workspace v0.1

三式超立方體四術驗證工程的乾淨工作區。

## 目的

把「能排、能解」推進到「可公開查證、可盲測、可重跑、可判失敗」。

本目錄不保存 61 串原始對話、私人命盤、身分資料或未授權個資。正式規則只以 `rule_id` 引用，原文留在受控來源庫。

## 四條工作線

1. **Dataset Audit（資料集稽核）**：檢查來源、授權、欄位、答案污染、出生資料可靠度與事件可核性。
2. **Blind Eval（盲測驗證）**：遮蔽答案，保存輸入、輸出、模型、prompt 版本、rule_id、分數與失敗原因。
3. **Prospective Cases（前瞻案例）**：六壬／奇門在結果發生前封存題目、時間、盤面、主判、應期與評分條件。
4. **Agentization（代理與技能化）**：驗證流程穩定後，再封裝成單一驗證代理；不先做多代理表演。

## Gate

- `READY`：來源與授權可核、輸入足夠、答案未洩漏、可盲測。
- `WARN`：可研究但存在流派、來源、時間精度或答案可靠度問題。
- `BLOCK`：來源不可核、答案污染、出生／問事時間缺失、無法客觀評分。

## 第一個可執行目標

先完成兩個公開倉庫的資料稽核，不立即跑全量模型：

1. 解析資料檔與欄位。
2. 產生來源清單與統計。
3. 抽取 10 題 pilot（試跑集）。
4. 將答案分離成封存鍵，執行端只看題本。
5. 由 5.6 執行；5.5 只看結果包與失敗證據，不接觸試題答案。

## 角色

- **雲龍**：最終裁決、封板、案例現實結果確認。
- **GPT-5.6**：資料稽核、工程、受控實算、失敗定位。
- **GPT-5.5／總司**：驗收設計、污染檢查、PASS/WARN/BLOCK 裁決建議。
- **Claude**：重大準度主張或規則衝突時第三方冷審。

## 禁止

- 不把 GitHub 題庫答案當命理真理。
- 不讓執行模型同時看到題本與答案鍵。
- 不因單案命中宣稱準度。
- 不把舊 Harness 骨架通過寫成真實驗證通過。
- 不把 61 串原文推到公開倉庫。
