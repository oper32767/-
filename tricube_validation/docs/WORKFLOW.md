# TriCube 四術驗證工作流 v0.1

## 0. 核心裁定

目前不再擴寫術理總綱。主線改為：

> 來源稽核 → 盲測封裝 → 受控實算 → 獨立評分 → 失敗回補 → 重跑驗證 → 達標後技能化

任何成果若沒有輸入、版本、答案隔離、執行紀錄與失敗條件，不算驗證。

---

## 1. 席位分工

### 雲龍｜最終裁決者

負責：

- 決定哪些 61 串條目屬正式封板規則。
- 判定現實事件結果與可接受的主判範圍。
- 裁決爭議案例的 PASS／WARN／BLOCK。
- 核准何時能宣稱某單術進入可用或商業測試。

雲龍不需要：

- 手動整理 GitHub JSON。
- 重複建立工程骨架。
- 替 AI 補齊來源或猜測欄位。

### GPT-5.6｜外部驗證與工程執行席

負責：

- 搜尋與稽核公開來源。
- 解析資料、正規化欄位、去重、標污染。
- 建立盲測題本、封存答案鍵與 hash manifest。
- 依 rule_id 做受控實算。
- 保存原始輸出、失敗原因、Unknown List 與 trace。
- 寫測試程式、資料 schema、GitHub Action 與後續代理原型。

限制：

- 不自行評定自己通過。
- 不把 dataset 答案當絕對真理。
- 不把未封板規則補成正式算法。

### GPT-5.5／總司｜治理與獨立驗收席

負責：

- 執行前審核 Gate 是否能客觀評分。
- 執行後只讀結果包、答案鍵與差異報告。
- 檢查答案污染、題目歧義、過寬命中與事後補話。
- 對 PASS／WARN／BLOCK 提出獨立裁決建議。
- 決定是否送 Claude 冷審。

5.5 不需要重做排盤或程式；若要複算，只抽查高承重錯誤案。

### Claude｜重大爭議冷審席

只在以下情況介入：

- 準度主張即將公開。
- 5.5 與 5.6 對案例有效性結論衝突。
- Dataset 本身答案可信度有重大疑問。
- 規則更改會影響多個既有測試結果。

## 2. 資料分桶

### 桶 A：公開 benchmark

用途：快速比較、固定題目、可重跑。

首批：

- BaziQA：八字選擇題。
- MingLi-Bench：八字＋紫微；可使用預排盤分離排盤與判讀。

限制：

- 題庫作者答案只稱 `dataset_ground_truth`。
- 未有獨立事件來源時，不得稱 `verified_reality_truth`。

### 桶 B：公眾人物歷史案例

用途：以獨立出生來源＋事件來源驗證靜態術。

必要欄位：

- 出生來源與可靠度。
- 事件來源與日期。
- 事件分類。
- 模型事前輸出。
- 不可事後修改的 hash。

### 桶 C：前瞻動態案例

用途：六壬、奇門。

必要欄位：

- 問題原文。
- 問事與起課／起局時間。
- 起課地點與時間口徑。
- 完整盤面。
- 主判、次判、應期、停機條件。
- 評分規則。
- 實際結果與證據。

## 3. 標準狀態機

`DISCOVERED` → `AUDITING` → `READY`／`WARN`／`BLOCK` → `SEALED` → `RUNNING` → `SCORED` → `REVIEWED` → `ACCEPTED`／`REJECTED`

規則：

- `BLOCK` 不進盲測。
- `WARN` 可研究，但不計正式準度。
- `SEALED` 後題本、答案鍵、prompt、rule set 均需 hash。
- `RUNNING` 後禁止修改題目與評分條件。
- 發現規則缺口，先登 Unknown；不可直接改答案。

## 4. 第一輪 Pilot

### Pilot-A｜BaziQA 10 題

目的：驗證資料稽核、答案隔離、rule_id 引用與結果記錄。

不追求準度宣稱。

### Pilot-B｜MingLi-Bench 10 題

分兩組：

1. `ASTRO_ON`：使用資料集預排盤，測判讀。
2. `ASTRO_OFF`：只給出生資料，測排盤＋判讀。

差值用於定位排盤誤差，不直接解釋為術數準度。

### Pilot-C｜六壬／奇門各 3 案

- 結果發生前封存。
- 期限以 1～7 日為主。
- 一案測正常判讀、一案測高歧義、一案測資料不足停機。

## 5. 5.5 Review Packet

每批交付固定包含：

1. `manifest.json`：檔案 hash、版本、模型、prompt、rule set。
2. `audit_report.json`：來源與資料問題。
3. `blind_booklet.json`：執行席可見。
4. `sealed_answer_key.json`：執行席不可見。
5. `raw_responses/`：逐題原始回答。
6. `score_report.json`：機械分數。
7. `failure_analysis.md`：錯誤分型與 Unknown。
8. `review_questions.md`：只列需 5.5 判斷的爭點。

5.5 不接受只有總分、沒有原始輸出的報告。

## 6. 能力 Gate

### V0｜流程可用

- 排盤核心錯誤 0。
- 答案隔離有效。
- 每題可回查 rule_id 與原始輸出。
- 缺資料能停機。

### V1｜單術內部可用

- 至少 30 個有效案例。
- 排盤準確率 ≥ 99%。
- 主判一致率 ≥ 85%。
- 無重大答案污染。
- 由 5.5 獨立驗收，雲龍終裁。

### V2｜外部測試可用

- 至少一個未公開 holdout。
- 跨模型重跑。
- 公開 benchmark 與現實案例分開報告。
- Claude 冷審高承重案例。

### V3｜商業測試

- 僅限已通過單術與已驗證題型。
- 明示不確定性與禁止用途。
- 完整保留 trace、同意、隱私與申訴機制。

---

## 7. 代理與技能化順序

不先建四個互相聊天的代理。

1. 先做一個 `ValidationAgent`：只負責稽核與執行測試。
2. 將資料正規化、hash、評分做成 deterministic tools（確定性工具）。
3. 將 61 串封板規則做成私有 Skill／file-search 來源，只回傳 rule_id 與必要片段。
4. 建立 guardrail：缺 T1／T2、答案鍵洩漏、未封板規則、私人資料外流即 BLOCK。
5. 穩定後才拆成 SourceAgent、ExecutionAgent、ReviewAgent。

---

## 8. 做不到與限制

- GPT-5.6 無法在這個對話內直接喚起另一個獨立 GPT-5.5 實例；5.5 必須由雲龍在其對話串接收 Review Packet。
- 執行 OpenAI Agents SDK 全自動測試需要 API key、可執行環境與費用額度；密鑰不得貼入對話或 GitHub。
- GitHub 公開倉庫不得放 61 串原文、私人命盤或可識別個資。
- 公開題庫可能已進入模型訓練資料，不能作唯一 holdout。
- 六壬、奇門目前缺成熟公開 benchmark，必須以持續前瞻封存補足。
