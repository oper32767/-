# 通用 AI 任務移交提示詞 v4.0｜風險審核與熱修補 v0.1

日期：2026-06-17
狀態：AI 初核修補建議，不是雲龍封板版。

## 1. 總判定

v4.0 不是完全壞掉，但存在高風險誘因：

```text
完整留存版過重。
來源標注過密。
禁止條款過多。
A4 要求完整成果全文，容易造成長篇重述。
AI 病歷雖然被隔離，但仍佔大量注意力。
最終輸出命令容易讓模型誤判：任何貼入此提示詞都要直接打包。
```

因此 v4.0 不宜作為唯一打包提示詞原版直接使用。

建議狀態：

```text
v4.0 暫停直接使用。
v4.0-hotfix 可作臨時修正版。
正式版應升 v4.1。
```

---

## 2. 主要風險

| 編號 | 風險 | 造成後果 | 修補方向 |
|---|---|---|---|
| RISK-001 | 完整留存版結構過重 | 模型容易重述歷史、拖長輸出 | 完整留存版改為索引＋成果清單；完整原文只附必要成果 |
| RISK-002 | A4 要求完整成果全文 | 會把一堆成果全文塞回去，造成上下文污染 | A4 改為「只列最新可用成果與路徑／檔名；全文另附附件或指定一件」 |
| RISK-003 | 來源標注要求過密 | 模型把每句話都變成標籤遊戲 | 僅成果、決策、規則、下一步需標注；背景摘要不逐句標 |
| RISK-004 | 禁止事項過多 | 壓過第一行動，造成模型怕錯而不施工 | 禁止事項縮到最多 10 條，短版最多 5 條 |
| RISK-005 | AI 病歷隔離篇幅過大 | 病歷雖隔離，仍佔注意力 | 病歷只保留 3 條以內「防再犯 Gate」 |
| RISK-006 | 最終輸出命令太強 | 使用者貼提示詞供審核時，模型可能誤打包 | 新增「只有使用者明確說打包／移交／換串才執行」 |
| RISK-007 | B 版不得超過 A 版 20% | A 很長時 B 也可能變長；比例不等於可用 | 改成 B 版硬上限：800～1200 字 |
| RISK-008 | 差異保全表可能過度負擔 | 沒有舊版也硬展開 | 無舊版時一行說明即可，不展開表 |
| RISK-009 | 不得開場白過硬 | 在審核、詢問、修補情境下不合用 | 僅限真正打包輸出時生效 |
| RISK-010 | 容易把打包當主任務 | 造成可執行資產產出下降 | 新增規則：打包只收斂，不新增工程主線 |

---

## 3. 熱修補條款

### HOTFIX-001｜執行情境 Gate

```text
只有使用者明確要求「打包」「移交」「換新串」「整理接續包」時，才執行本提示詞。
若使用者是在審核、修改、討論提示詞，AI 必須回應審核結果，不得直接打包。
```

### HOTFIX-002｜完整留存版降重

```text
完整留存版不再要求收錄所有成果全文。
只收錄：最新成果名稱、版本、檔名／路徑、用途、狀態、是否可直接用。
成果全文只放：使用者指定、短小、或下一串必須直接貼用者。
```

### HOTFIX-003｜短版硬上限

```text
新串接收短版硬上限 800～1200 字。
不得用「完整留存版 20%」作比例標準。
```

### HOTFIX-004｜第一動作優先

```text
新串短版第一動作必須放在前 1/3 篇幅內。
驗收錨點不得超過 5 條。
禁止誤判不得超過 5 條。
```

### HOTFIX-005｜病歷最小化

```text
AI 病歷不得在新串短版出現。
完整留存版最多保留 3 條防再犯 Gate。
不得寫長篇模型失衡史。
```

### HOTFIX-006｜可執行資產優先

```text
打包時必須列出「下一輪要產出的可執行資產」。
例如：CSV（逗號分隔表格）、JSON Schema（格式規格）、Validator（驗證器）、Skill.md（技能說明檔）、測試結果表。
不得只列整理、理解、補充、完善。
```

---

## 4. v4.1 修正方向

v4.1 應把打包提示詞拆成三段：

1. **執行情境判斷**：先判斷是否真的要打包。
2. **完整留存版**：只做索引、成果清單、缺口，不無限制貼全文。
3. **新串短版**：只留錨點、第一任務、第一動作、最小交付物、停機條件。

---

## 5. 目前裁定

```text
v4.0 原版：暫停直接使用。
v4.0-hotfix：可臨時使用。
v4.1：應另行改寫。
```

本檔只作審核與熱修補，不取代正式 v4.1。
