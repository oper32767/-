# 奇門主鏈接入與多引擎差分施工說明 v0.1

## 0. 這批施工解決什麼問題

這批檔案不是為了讓外部軟體取代六十一串，而是把「六十一串主規格」變成可檢查的機器設定，再讓不同引擎用同一份時間與口徑輸出結果。

核心目的有三個：

1. 找出排盤差異最早從哪一層開始。
2. 分清楚是曆法、定局、寄宮、暗干，還是解盤規則不同。
3. 阻止模型看到兩張不同的盤後直接猜哪張正確。

## 1. `qimen_internal_mainline_profile_v0.1.json`

用途：把六十一串已封板的奇門主鏈轉成機器可讀設定。

它鎖定：

- 時家、轉盤、置閏主鏈；
- 下北上南；
- 從零重建；
- 中五寄坤且寄星、寄干同步；
- 空亡、馬星、擊刑、入墓、門迫先列異常；
- 日干看求測人、時干看問事；
- 用神落宮整包；
- 暗干必須先聲明法別；
- 排盤未通過不得進解盤與運籌。

實際用途：它是內部主裁判的「規格輪廓」。外部引擎只能跟它比較，不能反過來覆蓋它。

## 2. `qimen_method_adapter_registry_v0.1.json`

用途：記錄每個引擎採用什麼方法。

例如：

- 拆補、置閏或茅山；
- 轉盤或飛盤；
- 晚子時同日或次日；
- 節氣來源；
- 天禽寄宮；
- 暗干法；
- 空亡與馬星算法。

實際用途：兩個引擎如果採不同方法，系統會先標示「方法不同」，而不是把結果差異誤判成程式錯誤。

## 3. `qimen_engine_output_schema_v0.1.json`

用途：規定所有引擎必須輸出相同欄位。

包含：

- 時間與政策；
- 四柱與節氣；
- 陰陽遁、局數、三元；
- 值符、值使；
- 天地盤、星、門、神；
- 暗干、空亡、馬星；
- 擊刑、入墓、門迫等異常。

實際用途：把不同軟體各自的輸出格式轉成同一語言，才有辦法逐欄比較。

## 4. `qimen_differential_case_set_v0.1.json`

用途：保存固定測試案例。

目前包含：

- 六組外部基準盤回放；
- 一組晚子時同日／次日雙政策測試；
- 節氣邊界測試標記。

實際用途：每次修改排盤程式後重跑同一批案例，可立即發現哪個欄位被改壞。

## 5. `tools/qimen_diff_harness.py`

`harness`（測試支架）在這裡指「把多個引擎輸出放到同一條檢查流程」。

用途：

1. 讀取兩個以上標準化 JSON（結構化資料）。
2. 按固定順序比較。
3. 回報第一個分歧與全部差異。
4. 不裁決哪一方正確。

比較順序從時間政策開始，再到節氣、四柱、局數、天地盤、星門神、暗干與異常。

實際用途：假設 A 引擎與 B 引擎最後八門不同，但真正第一個差異在「三元」。工具會指出三元才是源頭，不會只看到最後八門不同。

## 6. `tests/test_qimen_diff_harness.py`

用途：測試差分工具本身。

目前至少驗證：

- 完全相同時輸出 `PASS`（通過）；
- 局數不同時，第一分歧正確指出 `ju.number`（局數欄位）。

實際用途：避免負責抓錯的工具自己先出錯。

## 7. 目前不能做到的部分

這批施工已完成規格接入與差分工具，但內部六十一串排盤引擎本身尚未程式化。

所以目前可以：

- 接收不同引擎的標準化結果；
- 自動找第一差異；
- 驗證方法是否聲明完整；
- 保留來源、版本與源流。

目前還不能：

- 只輸入日期就由內部主鏈自動排出完整盤；
- 自動裁決外部引擎哪一個一定正確；
- 把外部解盤斷語直接升成正式規則。

下一階段是真正實作內部置閏轉盤排盤核心，再把外部引擎當差分參照。
